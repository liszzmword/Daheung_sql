## 1. 응답 속도 개선

### 1-1. 답변 스트리밍 (SSE)

**문제**: 질문하면 모든 처리(임베딩 → RAG 검색 → SQL 생성 → SQL 실행 → 자연어 답변 생성)가 끝난 뒤에야 답변이 한꺼번에 표시되어 10~20초 대기. 사용자 입장에서 "안 되는 거 아닌가?" 싶을 정도.

**해결**: API를 Server-Sent Events (SSE)로 전환.
- `meta` 이벤트: SQL 실행 완료 직후 SQL + 데이터 표를 즉시 전송 → 5초쯤에 표가 보임
- `delta` 이벤트: 자연어 답변을 글자 단위로 흘려서 표시
- `done` 이벤트: 완료 신호

총 응답 시간 자체는 비슷하나 사용자 체감은 2~3배 빨라짐.

**수정 파일**:
- `lib/sql.mjs` — `streamSQLAnswer` async generator 추가
- `lib/rag.mjs` — `streamRAGAnswer` 추가
- `api/query-sql.mjs` — SSE 응답으로 재작성
- `api/query.mjs` — SSE 응답으로 재작성
- `public/index.html` — SSE 파싱(`parseSSE`), `addStreamingMessage`, `finalizeStreamMessage` 추가

**커밋**: `4ffaa60 perf: 답변 스트리밍(SSE) 적용으로 체감 대기시간 단축`

### 1-2. Gemini 2.5-flash thinking 비활성화

**문제**: gemini-2.5-flash는 답변 생성 전에 내부적으로 "thinking" 단계를 거치는데, 단순 SQL 변환에는 불필요한데도 1.5~3초씩 잡아먹음.

**해결**: 모든 generateContent / generateContentStream 호출에 `config.thinkingConfig.thinkingBudget = 0` 적용.

**수정 파일**:

- `lib/rag.mjs` — generateRAGAnswer, streamRAGAnswer 2곳

**효과**: SQL 생성 + 답변 생성 양쪽에서 약 3~5초 절감.

## 2. 거래처 별칭 기능

### 2-1. 배경

영업 현장에서 거래처를 다양한 방식으로 부름:
- "주식회사 쓰리에이치" ↔ "쓰리에이치" ↔ "(주)쓰리에이치" ↔ "3H"

처음에는 customers 테이블에 `normalized_name` + `aliases` 컬럼을 추가해 LLM SQL 프롬프트에서 매칭하는 방식을 시도했으나, 매출 데이터(sales_clean) 중 `customer_code`가 NULL인 행이 많아 매칭에서 누락되는 문제 확인. 전체 롤백 후 더 안전한 방식으로 재설계.

### 2-2. 채택한 방식 — 질문 전처리

별칭 매칭을 LLM 프롬프트가 아닌 **사용자 질문 텍스트 전처리 단계**에서 처리.

```
사용자 질문: "2026년 3H 매출"
       ↓ (백엔드 전처리)
LLM 입력: "2026년 주식회사 쓰리에이치 매출"
       ↓ (기존 매칭 로직 그대로)
SQL 결과
```

**장점**:
- 기존 매칭 로직, LLM 프롬프트, customers/sales_clean 스키마 전혀 안 건드림
- 별칭 기능과 검색 로직이 독립적이라 별칭 미등록/오등록이 기존 검색에 영향 없음
- sales_clean.customer_code가 NULL인 행도 정상 매칭

### 2-3. 구현

**DB**: `customer_aliases` 테이블 신규 (마이그레이션은 Supabase SQL Editor에서 수동 실행)
- `alias` (UNIQUE, 항상 소문자)
- `canonical` (정식 명칭)

**백엔드**:
- `lib/aliases.mjs` — `normalizeAlias()`, `fetchAliases()`, `expandAliases()` 유틸
- `api/aliases.mjs` — GET/POST/DELETE CRUD
  - 중복 별칭 등록 시 409 Conflict + "이미 X에 매핑됨" 메시지
  - 모든 입력은 소문자 정규화 후 저장
- `api/query.mjs`, `api/query-sql.mjs` — LLM 호출 직전 `expandAliases()` 적용
- `expandAliases()`는 한글/영문/숫자 단어 경계 매칭 + 긴 별칭 우선 정렬로 `3HR`이 `3H`에 의해 깨지는 문제 방지

**프론트엔드** — 헤더 "별칭" 버튼 → 모달:
- 정식 명칭 입력 시 자동완성 (250ms 디바운스)
- `/api/customer-search` 신규 엔드포인트가 customers + sales_clean의 거래처명을 합쳐 distinct 반환
- 키보드 ↑↓ 탐색, Enter 선택, Esc 닫기, 마우스 클릭 모두 지원
- 입력 순서는 사용자 요청대로 정식명칭(좌) → 별칭(우)
- 칩 형태로 등록된 별칭 목록 표시, 삭제 버튼 개별 제공

**커밋**:
- `612b21c feat: 거래처 별칭 등록 + 질문 전처리`
- `a424dcf feat: 별칭 모달 입력 순서 교체 + 정식명칭 자동완성`

## 3. 보안 점검 (논의만, 미적용)

Supabase Table Editor에서 `purchases`, `sales_clean`, `sales_diary`, `query_logs`, `sync_log` 등에 `UNRESTRICTED` 표시 발견. RLS(Row Level Security) 미설정 상태.

현재 우리 앱은 `service_role_key`만 사용하므로 즉시 위험은 없음. 단, `anon_key`가 어떤 경로로든 노출되면 외부에서 직접 데이터 접근 가능. 방어선으로 RLS 활성화 권장.

```sql
-- 차후 적용 예정
ALTER TABLE purchases ENABLE ROW LEVEL SECURITY;
ALTER TABLE sales_clean ENABLE ROW LEVEL SECURITY;
ALTER TABLE sales_diary ENABLE ROW LEVEL SECURITY;
ALTER TABLE query_logs ENABLE ROW LEVEL SECURITY;
ALTER TABLE sync_log ENABLE ROW LEVEL SECURITY;
```

오늘은 보류, 별도 일정으로 진행 예정.

## 4. 시도했다가 롤백한 작업 (참고용)

오전~점심 시간대에 customers 테이블에 `normalized_name` + `aliases` 컬럼을 직접 추가하고 LLM SQL 프롬프트를 customer_code 기반 매칭으로 전면 교체하는 작업을 진행했음.

**롤백 사유**:
- 매출 데이터 중 customer_code가 NULL인 row가 많음 → 새 매칭 패턴이 IN 절로 customer_code를 거름 → NULL인 row는 검색 결과에서 누락
- LLM이 다중 CTE를 빈 줄 포함 포맷팅 → cleanSQL 함수의 정규식 버그가 빈 줄 이후를 잘라먹음 → 깨진 SQL 생성

git revert로 코드 복원 (커밋 `e7b7624`, `9f52abb`), customers 테이블 컬럼은 사용자가 수동 DROP으로 제거.

이 경험을 바탕으로 오후 작업은 **기존 매칭 로직 안 건드리고 사이드카로 별칭 기능만 추가**하는 방향으로 재설계.
